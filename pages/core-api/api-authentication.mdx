# Quotas & Limits

This page explains how rate limiting and quota enforcement work for the Safe Infrastructure API.

## Authentication

All sustained API usage requires authentication. Include your API key in the `Authorization` header:
```bash
curl -X GET "https://api.safe.global/tx-service/eth/api/v2/safes/0x..." \
  -H "Authorization: Bearer YOUR_API_KEY"
```

Generate your API key at the [developer dashboard ↗](https://developer.safe.global). For step-by-step instructions, see [API Keys & Authentication](./how-to-use-api-keys).

<Callout type="warning">
  **Deprecation notice:** Public (unauthenticated) API endpoints will be deprecated. Transition to authenticated access as soon as possible to prevent disruptions.
</Callout>

### Why authenticate?

- **Higher rate limits**: Authenticated requests receive significantly higher RPS allocations than unauthenticated requests.
- **Quota tracking**: Your usage is tracked per account, enabling clear visibility into consumption.
- **Reliability**: Authenticated traffic is prioritized and protected from noisy-neighbor effects.

## API unit definition

All usage is measured in **API units**:

> **1 API unit = 1 HTTP request OR 1 webhook delivery**

## Enforcement behavior

Quota enforcement follows a progressive approach designed to give you time to respond before hard limits take effect.

### 1. Warning — 80–90% of monthly quota

When your account reaches 80–90% of its monthly API unit quota:

- A warning notification is sent to the account owner (email and dashboard).
- API access continues at full RPS.
- No degradation occurs at this stage.

### 2. Soft cap — 100% of monthly quota

When your account reaches 100% of its monthly quota:

- Your effective RPS is **reduced** (throttled).
- Requests continue to be served, but at a lower rate.
- You are notified that you have reached your quota.

### 3. Hard cap — after grace period

If usage continues beyond the soft cap grace period:

- Requests exceeding the limit receive an **HTTP 429 Too Many Requests** response.
- The response includes a `Retry-After` header indicating when to retry.
- Your integration should implement exponential backoff or circuit-breaker logic.

Example 429 response:
```json
{
  "detail": "Request was throttled. Expected available in 30 seconds."
}
```

## Account-level enforcement

Enforcement is applied at the **account level**, not per API key. This means:

- All API keys under a single account share the same quota and RPS allocation.
- Creating multiple API keys does not increase your limits.
- Usage from all keys is aggregated for enforcement purposes.

<Callout type="warning">
  **Anti-circumvention:** Creating multiple accounts to circumvent rate limits or quotas violates the Safe Infrastructure terms of service. Accounts engaging in this behavior may be suspended.
</Callout>

## Handling rate limit errors

When you receive an HTTP 429 response:

1. **Read the `Retry-After` header** to determine when to retry.
2. **Implement exponential backoff** — wait progressively longer between retries.
3. **Queue non-urgent requests** — batch or defer requests that are not time-sensitive.
4. **Monitor your usage** via the [developer dashboard](https://developer.safe.global).

### Example: handling 429 responses
```typescript
async function fetchWithRetry(
  url: string,
  options: RequestInit,
  maxRetries = 3
) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status === 429) {
      const retryAfter = parseInt(
        response.headers.get('Retry-After') || '30',
        10
      );
      const backoff = retryAfter * 1000 * Math.pow(2, attempt);
      await new Promise(resolve => setTimeout(resolve, backoff));
      continue;
    }

    return response;
  }

  throw new Error('Max retries exceeded');
}
```

## Security best practices

- **Store keys securely**: Use environment variables or a secret manager. Never hard-code API keys.
- **Rotate keys periodically**: Generate new keys on a regular schedule and revoke old ones.
- **Revoke compromised keys immediately**: If a key is leaked, revoke it in the dashboard and generate a new one.
- **Use separate keys per environment**: Maintain different keys for development, staging, and production.
